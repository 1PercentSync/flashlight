# DeepSeek 缓存机制研究

日期：2026-05-05
模型：deepseek-v4-flash

---

## 确认的机制

每次请求完成后，在以下位置创建一个缓存单元：

```
缓存单元位置 = ⌊total_prompt_tokens / 128⌋ × 128
```

后续请求命中该单元的条件：
1. 新请求的 `total_prompt_tokens` ≥ 缓存单元大小
2. 新请求的前 N 个 tokens 与缓存单元内容**完全相同**

`prompt_cache_hit_tokens` 始终是 128 的整数倍。多个单元可匹配时取最大的。

`total_prompt_tokens` < 128 时不创建缓存单元（向下取整为 0）。

---

## 核心特性

| 特性 | 行为 |
|------|------|
| 最小大小 | total < 128 tokens → 无缓存 |
| 粒度 | 始终 128 对齐 |
| 跨 key 碰撞 | 不可能（不同文本 = 不同 tokens） |
| 方向性 | 短→长 命中；长→短 不命中 |
| 时效性 | 请求完成后立即可用 |
| 驱逐 | LRU，TTL 实测约 6 小时 |

---

## 方向性规则解释

- 请求 A (543 tok) 创建单元在 512
- 请求 B (864 tok) 到达：864 ≥ 512，前 512 tokens 匹配 → **hit=512**
- 反过来：A (864 tok) 创建单元在 768。B (543 tok)：543 < 768 → **hit=0**

短请求够不到长请求的缓存单元。长请求能够到短请求的缓存单元。

## 测试证据

### 测试 1：方向性

```
长(864) → 短(567):  hit=0    (567 < 768)
短(567) → 长(864):  hit=512  (864 ≥ 512)
短(567) → 短(567):  hit=512  (567 ≥ 512，相同前缀)
```

### 测试 2：最低门槛

```
total=24:   hit=0  (未创建单元)
total=81:   hit=0  (未创建单元)
total=320:  hit=256
total=742:  hit=640
```

### 测试 3：无碰撞

```
Key A 已建立 → Key C 首次探测: hit=0
（完全不同的前缀文本，不会互相命中）
```

### 测试 4：生产场景模拟

```
查询1(864) → 探测1(567): hit=0     (探测够不到查询的 768 单元)
探测1(567) → 查询2(866): hit=768   (查询够到了查询1 的 768 单元)
查询2(866) → 探测2(567): hit=512   (探测够到了探测1 的 512 单元)
```

### 测试 5：激活场景

```
查询(864) → 激活(565): hit=0    (激活够不到 768)
激活(565) → 探测(567): hit=512  (探测够到了激活的 512 单元)
```

## 对 Flashlight 的影响

### 为什么探测无法直接命中查询的缓存

查询创建单元在 ~⌊50000/128⌋×128 ≈ 49920。探测约 567 tokens。567 < 49920 → 永远不命中。

### 为什么下次查询无法命中上次查询的缓存（changes 不同）

查询 1：[firstTurn, base, changes1, query1] → 单元在 49920，内容包含 changes1。
查询 2：[firstTurn, base, changes2, query2] → 前 49920 tokens 在 changes 处分叉 → 不匹配。

### 解决方案：重建后发两次激活

| 激活 | 创建的单元 | 受益方 |
|------|-----------|--------|
| `[firstTurn, "OK"]` (~565 tok) | ~512 | 探测 (567 ≥ 512) ✓ |
| `[firstTurn, base, "OK"]` (~48000 tok) | ~47872 | 查询 (50000 ≥ 47872) ✓ |

激活后：
- 每次探测命中 512（确认缓存存活）
- 每次查询命中 ~47872（节省 base 部分的费用）

### 成本

- 短激活：~565 tokens × ¥1/百万 = ¥0.000565（可忽略）
- Base 激活：~48000 tokens × ¥1/百万 = ¥0.048（重建后一次性）
- 每次查询节省：~48000 × ¥0.98/百万 = ¥0.047
- 回本点：重建后第 2 次查询

---

## 对设计的约束

1. `firstTurn` 必须足够长，使探测 total > 128（否则永远无法创建缓存单元）
2. `cache_key` 必须在 `firstTurn` 最开头（确保分叉在 token 3~5 处，远低于 128）
3. 激活仅在重建后发送（不需要每次查询后发）
4. 激活失败不影响系统（下次探测 miss → 重建 → 自愈）

## 未知 / 风险

1. **机制是逆向工程得出的。** DS 没有公开保证该行为稳定。服务端更新可能静默破坏缓存策略。
2. **驱逐策略不确定。** 我们假设按最后访问时间 LRU。实际策略可能不同。
3. **128 token 块对齐是观察结果，非官方文档。** 可能变化。

缓解：可选的启动验证测试，检测机制是否仍然生效。
